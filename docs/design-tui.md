OpenCode 作为一个广受欢迎的开源 AI 编程助手（常被视为 Cursor 或 Claude Code 的开源及本地化替代方案），其 TUI（终端用户界面）设计非常注重**开发者体验（DX）**、**键盘驱动效率**以及**上下文管理的直观性**。

以下是 OpenCode TUI 设计的详细拆解：

### 1. 核心交互逻辑与语法糖
OpenCode 的 TUI 围绕命令行输入框构建，并通过几个特殊符号极大简化了上下文的引入和操作：
* **文件引用 (`@`)**：在输入框中输入 `@` 会唤起当前工作目录的模糊搜索。选中文件后，该文件的内容会自动作为上下文附加到对话中（例如：`@src/api/index.ts 这里的鉴权是如何处理的？`）。
* **Shell 集成 (`!`)**：以 `!` 开头的消息会直接作为 Bash/Shell 命令执行（例如：`!ls -la` 或 `!npm run test`）。命令的终端输出会被自动捕获，并作为工具执行结果提交给 LLM，方便 AI 根据报错信息进行 Debug。
* **斜杠命令 (`/`)**：输入 `/` 可以快速触发内置的系统操作。例如 `/help`（查看帮助）、`/compact` 或 `/summarize`（压缩当前会话以节省 Token）、`/models`（切换模型）、`/export`（导出对话为 Markdown）等。

### 2. 双模式工作流 (Plan vs. Build)
为了防止 AI 代理“暴走”或在没有清晰思路前乱改代码，TUI 设计了两种核心状态（可通过 `Tab` 键快速切换，状态指示器位于右下角）：
* **Plan Mode（计划模式）**：只读模式。LLM 不会修改任何文件，而是通过对话提出架构建议或代码实现方案。
* **Build Mode（构建模式）**：执行模式。LLM 获准使用文件编辑工具，直接对代码库进行修改。

### 3. 视觉设计与渲染

OpenCode 在终端中实现了类似现代 GUI 编辑器的视觉体验：
* **真彩色（Truecolor）支持与主题**：TUI 充分利用 24-bit 真彩色。除了能自动适配终端背景颜色的 `system` 主题外，还内置了开发者喜爱的经典配色（如 `catppuccin`, `tokyonight`, `nord`, `gruvbox` 等）。
* **差异视图（Diff Rendering）**：当 AI 修改代码时，TUI 会实时渲染语法高亮的 Diff 视图。通过 `diff_style` 配置，用户可以选择 `auto`（根据终端宽度自适应展现）或 `stacked`（单列堆叠显示）。
* **工具执行可视化**：TUI 会清晰地折叠或展开 LLM 的工具调用过程（如读取文件、执行搜索、运行命令等），用户可以通过 `/details` 命令切换底层执行细节的可见性。
* **平滑滚动**：支持 macOS 风格的滚动加速（`scroll_acceleration.enabled`），在快速滚动长代码或长对话时提供自然、平滑的视觉反馈。

### 4. 极致的键盘驱动 (Leader Key 机制)
为了让极客用户手不离键盘，TUI 引入了类似 Tmux 逻辑的**前导键（Leader Key）**机制：
* 默认前导键为 `ctrl+x`。
* 按下前导键后接单字母即可触发快捷操作，例如：`ctrl+x c`（压缩会话）、`ctrl+x q`（退出会话）、`ctrl+x e`（调用外部编辑器）。
* **外部编辑器集成**：当终端底部输入框不足以编写长篇复杂的 Prompt 时，按下 `ctrl+x e` 会自动唤起 `EDITOR` 环境变量指定的外部编辑器（如 Vim, Nano 或 VS Code），保存并关闭后，内容会自动同步回 TUI 的输入框。

### 5. 现代化交互增强
* **多模态拖拽支持**：可以直接将图片拖拽进终端窗口，TUI 会自动解析路径并将图片添加到 Prompt 中，供支持多模态的大模型进行视觉分析。
* **多会话管理**：通过 `/sessions`（或 `ctrl+x l`），用户可以在终端内以列表形式查看、恢复或在之前的历史对话之间进行无缝切换。

深入探讨 TUI（终端用户界面）的交互逻辑，本质上是在剖析它的**状态管理**、**异步事件流**以及**焦点控制**。一个优秀的 AI TUI 不能只是简单的“请求-等待-打印”的死循环，它必须是**非阻塞（Non-blocking）**且**高响应**的。

以下是 OpenCode TUI 在实际运行时的完整交互逻辑与底层机制拆解：

### 1. 焦点流转与层级管理 (Focus & Z-Index Logic)
在纯文本的终端中没有真正的鼠标指针，所有的交互都围绕**焦点（Focus）**展开。
* **输入框驻留（Input Trapping）**：在绝大多数状态下，用户的键盘输入焦点被强制锁定在底部的输入框中。即使用户使用鼠标滚轮或快捷键向上翻阅历史对话，焦点依然在输入框，直接打字不会导致界面跳回底部或输入错乱。
* **悬浮层劫持（Modal Overlays）**：当触发 `@`（文件搜索）或 `/`（命令菜单）时，TUI 会动态渲染一个“悬浮层”。此时，键盘焦点被**临时劫持**到悬浮层。
    * **交互表现**：此时按下 `↑` / `↓` 不再是翻阅历史记录，而是变为在搜索列表中上下移动高亮光标。
    * **退出逻辑**：按下 `Enter`（确认选择）或 `Esc`（取消操作）后，悬浮层销毁，焦点瞬间交还给主输入框，保证输入流的连续性。

### 2. 异步流式渲染与中断机制 (Async Streaming & Interruption)
由于大模型生成内容需要时间，TUI 必须优雅地处理“等待”状态。
* **流式打字机效果**：当发送 Prompt 后，TUI 进入**生成态**。此时输入框通常会变为“不可用”状态（或变为一个取消按钮提示）。大模型返回的 Token 碎片会被异步推送到 UI 渲染线程，形成平滑的打字机输出效果，同时实时解析 Markdown 语法（例如遇到 ` ``` ` 会瞬间闭合一个代码块并应用语法高亮）。
* **精准中断 (SIGINT 捕获)**：在生成态中，如果用户按下 `Ctrl+C`，TUI 不会直接崩溃退出程序。它的逻辑是：
    1. 捕获 `SIGINT` 信号。
    2. 向底层的 LLM Worker 线程发送中断指令，切断网络请求。
    3. 在视图区留下一个类似 `[已中断]` 的标识。
    4. 立刻将焦点和控制权还给底部的输入框，让用户可以重新输入。

### 3. 上下文组装与“芯片化”交互 (Context Assembly & Tokenization)
这是 OpenCode 区别于普通终端脚本的杀手锏。
* **Token 化标签（Chips）**：当用户通过 `@` 选中一个文件（如 `@src/utils.js`）时，这个字符串在输入框中不再是纯文本，而是被渲染成一个具有独立颜色的**“芯片（Chip/Tag）”**。
    * **交互逻辑**：此时按下 `Backspace` 退格键，不会只删掉一个字母 `s`，而是将整个 `@src/utils.js` 芯片一次性删除。这种逻辑极大提升了修改长文件路径时的效率。
* **隐式上下文挂载**：在 UI 层面，您只看到了一个带有文件名的标签；但在底层状态机中，TUI 已经读取了该文件的完整文本，并将其放入了待发送给大模型的 `system_prompt` 或上下文数组中。

### 4. Diff 视图与代码审查闭环 (Diff Review Loop)
在 **Build（构建）模式** 下，当 AI 决定修改文件时，TUI 会进入一个特殊的代码审查交互流：
1. **拦截与解析**：AI 输出标准格式的修改指令（如 Search/Replace 块或 unified diff）。TUI 解析这些指令，但**暂不写入磁盘**。
2. **差异渲染**：在主视图区渲染红绿对比的 Diff 视图（删除行为红底，新增行为绿底）。
3. **等待确认 (Wait for Confirmation)**：如果系统开启了安全审查（默认行为），界面会暂停，并在输入框上方弹出交互提示：`[Y] 接受修改 (Accept) | [N] 拒绝 (Reject) | [E] 编辑 (Edit)`。
    * 选 `Y`：底层触发 I/O 操作，真正修改本地文件，并打印“修改已应用”。
    * 选 `N`：丢弃修改，向 AI 反馈“用户拒绝了此修改”。
    * 选 `E`：针对复杂的修改，调用外部编辑器让用户手动微调这段 Diff。

### 5. 工具链执行的“手风琴”逻辑 (Accordion UI for Tools)
当 AI 需要调用外部工具（如运行 `npm test` 或读取长日志）时，直接打印全量日志会严重破坏对话体验。
* **状态折叠**：TUI 采用类似“手风琴”的折叠逻辑。开始执行工具时，渲染一行带 Loading 动画的占位符：`⠋ 正在执行: npm test`。
* **动态更新**：如果命令执行成功，占位符变为绿色：`✔ 执行完成: npm test`；如果失败则变为红色：`✖ 执行失败: npm test (Exit code 1)`。
* **按需展开**：用户想要看报错详情时，通过输入 `/details` 或使用快捷键，该占位符向下展开，吐出带有 ansi 颜色的终端原始 Stdout/Stderr 输出。

### 6. 历史命令漫游 (History Roaming)
借鉴了 Bash/Zsh 的交互习惯：
* 当输入框为**空**时，按下 `↑` 键，TUI 会从内存的历史栈（History Stack）中调出您上一次输入的完整 Prompt。
* 您可以继续按 `↑` 向前追溯，或按 `↓` 向后返回。
* 一旦您对历史 Prompt 进行了修改，或者输入框已有新内容，`↑` 和 `↓` 键的光标移动逻辑将自动切换回**多行文本内的上下移动**。这种上下文感知的按键复用（Key Overloading）是顶级 TUI 的标志。
